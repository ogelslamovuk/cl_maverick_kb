# Maverick KB — описание проекта для нейросетки

## Что это за проект

**Maverick KB** — локальная база знаний по экосистеме Maverick. Цель проекта — превращать исходные материалы по продукту и операциям клиента в понятные рабочие инструкции для сотрудников.

База нужна для:

- онбординга нового клиента;
- обучения сотрудников кинотеатра/объекта;
- поддержки кассиров, администраторов, бухгалтерии и технических специалистов;
- подготовки инструкций, FAQ, чеклистов, сценариев обучения и ответов ассистента.

Главная единица результата — не пересказ источника, а **практическая инструкция**: что делать, где нажать, что проверить, какие риски учесть, когда передавать дальше.

## Какие материалы обрабатываются

Входом могут быть:

- видео и screencast-записи;
- аудио;
- транскрипты;
- скриншоты;
- PDF/DOCX/TXT;
- заметки из чата;
- объяснения эксперта;
- поддержка/инциденты;
- существующие wiki-страницы.

Для видео аудио и визуальная часть считаются одной связанной единицей: экран показывает состояние интерфейса, голос объясняет смысл действия, бизнес-правила и исключения. Нельзя делать инструкцию только по картинкам или только по тексту, если в материале есть и то и другое.

## Что считается хорошей статьёй

Хорошая статья:

- решает конкретную рабочую задачу;
- начинается с цели и ситуации применения;
- говорит, для кого она написана;
- даёт порядок действий;
- объясняет проверку результата;
- предупреждает о рисках денег, НДС, оплат, возвратов, сертификатов, персональных данных и технической настройки;
- содержит полезные скриншоты рядом с шагами;
- содержит связанные страницы;
- не содержит мусора, пересказа, внутренней кухни и следов неуверенности модели.

Плохая статья:

- пересказывает видео;
- перечисляет экраны без задачи;
- дублирует уже существующую страницу;
- пишет неподтверждённые выводы как факты;
- оставляет `TODO`, `QUESTIONS`, `нужно уточнить`, `по видео видно`, `в источнике сказано` в публичном тексте;
- создаёт новую страницу вместо обновления существующей.

## Основная логика навигации

Навигация строится вокруг рабочих задач и контекста пользователя, а не вокруг внутренней архитектуры продукта.

Основные разделы:

- `Билеты и касса`;
- `Manager / back-office`;
- `Зал и ресторан`;
- `Киоск`;
- `Афиша и расписание`;
- `Настройки и налоги`;
- `Поддержка`.

Если тема относится к нескольким разделам, создаётся одна каноническая страница и ссылки из вторичных разделов. Две полные дублирующие инструкции вести нельзя.

## Важные правила проекта

1. Не выдумывать факты.
2. Не публиковать сырьё напрямую.
3. Не создавать новую страницу, пока не проверены существующие страницы.
4. Если тема уже есть — обогатить и уточнить каноническую страницу, а не делать дубль.
5. Противоречия не сглаживать. Если нельзя решить по источникам — вынести вопрос в `QUESTIONS.md`.
6. Публичная страница должна быть инструкцией для пользователя, а не отчётом об обработке материала.
7. Каждое локальное изменение wiki должно быть синхронизировано в GitHub Pages, если задача касается опубликованной базы.
8. Готово только после build, проверки ссылок/картинок, commit, push, deploy и live-проверки.
