# ТЗ: создание инструкций Maverick KB из исходных материалов

## 1. Назначение

Это ТЗ описывает, какой результат должна выдавать нейросетка/агент при обработке материалов для Maverick KB.

Задача агента — превратить видео, скриншоты, транскрипты, документы или заметки в рабочие инструкции для базы знаний Maverick.

## 2. Входные данные

На вход может быть передано одно или несколько:

- папка с видео;
- отдельное видео;
- аудио;
- скриншоты;
- PDF/DOCX/TXT;
- транскрипт;
- заметки эксперта;
- существующая wiki-страница;
- ссылка на материал.

Пользователь должен указать путь к материалам или явно передать файлы.

## 3. Ожидаемый результат

Агент должен создать или обновить документы Maverick KB:

1. Рабочие wiki-страницы в Obsidian vault.
2. Публичные страницы в MkDocs/GitHub Pages, если задача затрагивает опубликованную базу.
3. Скриншоты/изображения для инструкции.
4. Вопросы по неподтверждённым фактам в `QUESTIONS.md`.
5. Запись в `CHANGELOG.md`.
6. Обновление индекса/навигации, если добавлены новые страницы.

## 4. Основной критерий качества

Итоговая страница должна помогать сотруднику выполнить задачу без обращения к исходному видео.

Страница должна отвечать:

- что это за задача;
- кому нужна инструкция;
- когда её применять;
- что нужно до начала;
- какие шаги выполнить;
- как проверить результат;
- какие есть риски;
- какие ошибки бывают;
- куда перейти дальше.

## 5. Что не является результатом

Не считается готовой работой:

- сырой транскрипт;
- краткий пересказ видео;
- список экранов без задачи;
- набор скриншотов без инструкции;
- локально изменённые файлы без публикации;
- build без live-проверки;
- статья с неподтверждёнными фактами;
- статья с внутренними маркерами `TODO`, `QUESTIONS`, `high-risk`, `verified`.

## 6. Обязательная процедура

### 6.1. План перед стартом

Перед обработкой агент должен дать пользователю краткий план и дождаться явного разрешения.

План должен содержать:

- ожидаемый результат;
- какие материалы будут обработаны;
- какие страницы/разделы могут измениться;
- как будет проверяться результат.

До разрешения нельзя:

- транскрибировать;
- извлекать кадры;
- редактировать wiki;
- собирать сайт;
- коммитить;
- пушить.

### 6.1.1. Автономный режим

Если пользователь дал стартовое сообщение из `docs/MAVERICK_KB_INITIAL_MESSAGE.md`, оно уже является разрешением на полный цикл. В этом режиме агент не ждёт отдельного `approve` и обязан автономно довести задачу до готовой инструкции, публикации и live-проверки.

Остановка на этапе транскрипции запрещена, если причина только в том, что команда `whisper` не найдена в PATH, не задан `OPENAI_API_KEY` или не установлен Python-модуль `openai`. Проект использует локальный Windows Whisper по прямому пути `D:\soft\Whisper\whisper-cli.exe`.

### 6.2. Анализ источника

Агент должен:

- проверить существование исходников;
- составить список файлов;
- извлечь/прочитать текст;
- для видео получить транскрипт и визуальные опоры;
- выделить подтверждённые факты;
- отделить вопросы и спорные места.

### 6.2.1. Транскрипция: обязательные fallback-и

Для видео сначала проверяй прямой локальный путь:

```text
D:\soft\Whisper\whisper-cli.exe
```

В Git Bash/MSYS используй:

```bash
/d/soft/Whisper/whisper-cli.exe
```

Модели:

```text
D:\soft\Whisper\models\ggml-base.bin
D:\soft\Whisper\models\ggml-small.bin
D:\soft\Whisper\models\ggml-large-v3.bin
```

Если `whisper` не найден как команда, это не ошибка. Команда `whisper` в PATH не требуется.

Fallback-порядок:

1. `whisper-cli.exe` по прямому пути + `ggml-base.bin` или `ggml-small.bin`.
2. Нарезка длинного аудио на сегменты и транскрипция сегментов.
3. `D:\JetBrains\cl_maverick_kb\.venv_stt` / `faster-whisper`, если доступен.
4. Визуальный разбор по contact sheets/скриншотам + частичный транскрипт; спорное — в `QUESTIONS.md`.

Отсутствие `OPENAI_API_KEY` не блокирует задачу.

### 6.3. Анализ текущей базы

Агент должен проверить существующие страницы и решить:

- обновить существующую страницу;
- создать новую страницу;
- объединить дубли;
- вынести материал в вопросы;
- не публиковать, если фактов недостаточно.

Приоритет — обновлять канонические страницы, а не плодить новые.

### 6.4. Написание инструкции

Страница пишется как пользовательская инструкция.

Обязательные элементы, если применимы:

- цель;
- аудитория;
- ситуация применения;
- ожидаемый результат;
- путь в интерфейсе;
- шаги;
- контроль результата;
- риски;
- частые ошибки;
- связанные страницы.

Пустые разделы удаляются.

### 6.5. Работа с вопросами

Если факт не подтверждён, агент обязан вынести его в:

```text
D:\JetBrains\cl_maverick_kb\Maverick_KB\QUESTIONS.md
```

Формат вопроса:

```md
- [[Раздел/Страница]]: точный вопрос?
```

Нельзя заменять неизвестные факты фразами `возможно`, `скорее всего`, `нужно уточнить` в публичной статье.

### 6.6. Синхронизация публикации

Если изменена рабочая wiki-страница, нужно синхронизировать публичную Pages-версию:

- обновить `D:\JetBrains\cl_maverick_kb_pages\docs`;
- скопировать картинки в `docs/assets/media`;
- конвертировать Obsidian-ссылки и картинки;
- обновить `mkdocs.yml`, если изменилась навигация.

### 6.7. Проверка и публикация

Обязательно:

```bash
cd /d/JetBrains/cl_maverick_kb_pages
uvx --with mkdocs-material mkdocs build --strict
```

Затем проверить:

- ссылки;
- картинки;
- search index;
- визуальный вид страницы;
- git status;
- commit;
- push;
- GitHub Pages deploy;
- live URL.

## 7. Требования к стилю

- Русский язык.
- Коротко и по делу.
- Пользовательская задача важнее внутренней архитектуры продукта.
- Никакого маркетинга.
- Никакого сырого конспекта.
- Никаких неподтверждённых фактов.
- Для рискованных тем — явные предупреждения.

## 8. Рискованные темы

Высокий риск:

- деньги;
- оплаты;
- возвраты;
- сертификаты;
- НДС;
- бухгалтерия;
- персональные данные;
- production-настройки;
- технические службы и интеграции.

По таким темам нельзя выдумывать правила. Если источник не даёт точного ответа, вопрос уходит в `QUESTIONS.md`.

## 9. Требования к скриншотам

Скриншоты должны:

- поддерживать конкретный шаг;
- быть читаемыми;
- не содержать лишних окон, таскбара, панели встречи;
- сохранять структуру таблиц и форм;
- не быть чрезмерно замазанными.

Если источник — training/STAGE, данные считаются фейковыми, если пользователь не сказал обратное. В этом случае не надо редактировать скриншоты по умолчанию.

## 10. Приёмка результата

Работа принимается, если агент может показать:

- список изменённых страниц;
- пути к файлам;
- какие вопросы добавлены;
- результат `mkdocs build --strict`;
- результат проверки ссылок/картинок;
- commit hash;
- статус GitHub Pages deploy;
- live URL.

## 11. Минимальный финальный отчёт

```text
Готово.

Создано/обновлено:
- ...

Где лежит:
- ...

Проверка:
- mkdocs build --strict: OK
- links/assets: OK
- deploy: success
- live URL: ...

Вопросы:
- ...
```
