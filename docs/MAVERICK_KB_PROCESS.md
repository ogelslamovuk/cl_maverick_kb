# Maverick KB — процедура создания инструкции

## Ожидаемый результат обработки

После обработки материала должны появиться или обновиться рабочие инструкции в Maverick KB.

Результат считается готовым, если:

- материал разобран по смыслу, а не пересказан;
- существующие страницы проверены на пересечение;
- новая информация влита в канонические страницы без дублей;
- публичный текст написан как инструкция для пользователя;
- неизвестные факты вынесены в `QUESTIONS.md`;
- vault и GitHub Pages синхронизированы;
- сайт собран, опубликован и проверен live.

## Общий порядок работы

### 1. Принять задачу и источник

Определи:

- где лежат исходные материалы;
- что это за материалы: видео, аудио, скриншоты, текст, PDF, заметки;
- какой продукт/раздел затронут;
- какая роль пользователя выполняет действие;
- какая рабочая задача должна быть описана;
- есть ли риски: деньги, оплаты, возвраты, НДС, сертификаты, персональные данные, техническая настройка.

Если источник не найден — не продолжай. Сообщи точный путь, который проверялся.

### 2. Перед стартом дать план и дождаться разрешения

Для больших задач по Maverick KB сначала дай короткий план:

- ожидаемый результат;
- какие материалы будут обработаны;
- какие страницы, вероятно, будут затронуты;
- как будет проверяться результат.

После плана остановись и жди `approve`, `давай`, `ок` или другого явного разрешения.

Не начинай транскрипцию, извлечение кадров, редактирование wiki, build, commit или push до одобрения.


## Автономный режим без повторного approve

Если пользователь дал initial message из `docs/MAVERICK_KB_INITIAL_MESSAGE.md`, это уже считается явным разрешением на полный цикл обработки. В этом режиме агент не должен останавливаться после плана и ждать дополнительного `approve`.

В автономном режиме агент сам выполняет весь цикл:

1. проверяет исходную папку;
2. инвентаризирует материалы;
3. извлекает аудио/кадры;
4. делает транскрипцию через локальный Windows Whisper по прямому пути;
5. при сбоях применяет fallback-и;
6. пишет/обновляет wiki;
7. обновляет `QUESTIONS.md`, `INDEX.md`, `CHANGELOG.md`;
8. синхронизирует Pages;
9. запускает build;
10. проверяет ссылки, картинки, поиск;
11. делает commit/push;
12. ждёт GitHub Pages deploy;
13. проверяет live URL;
14. только потом отвечает пользователю.

Остановка допустима только если:

- исходная папка физически недоступна;
- агент запущен не на той машине и не видит `D:\JetBrains` / `D:\soft`;
- нет прав на запись в проект;
- команда может уничтожить данные или требует явного выбора пользователя.

Отсутствие команды `whisper` в PATH, отсутствие `OPENAI_API_KEY`, отсутствие Python-модуля `openai` или отсутствие `faster_whisper` не являются причинами остановки.

### 3. Инвентаризировать материалы

Для папки с видео/аудио:

- составь список файлов;
- проверь длительность;
- зафиксируй порядок;
- пойми, какие файлы относятся к одной теме;
- создай рабочую папку под извлечённые материалы, если её ещё нет.

Обычно рабочие извлечения кладутся в:

```text
Maverick_KB/media/_video_extracted/<topic-slug>/
```

### 4. Получить транскрипт и визуальные опоры

Для видео нужны:

- транскрипт речи;
- ключевые кадры или contact sheet;
- отдельные скриншоты для публикации;
- сопоставление аудио и экрана.

Проверенный локальный инструмент транскрипции:

```text
D:\soft\Whisper\whisper-cli.exe
```

Модели:

```text
D:\soft\Whisper\models\ggml-base.bin
D:\soft\Whisper\models\ggml-small.bin
D:\soft\Whisper\models\ggml-large-v3.bin
```

Для длинных видео лучше сначала нарезать аудио на сегменты и транскрибировать сегменты, а не пытаться обработать весь файл целиком.

Если транскрипт плохой, нельзя слепо использовать галлюцинации. Нужно сверять с экраном, повторить важные фрагменты другой моделью/сегментами или вынести спорное в вопросы.


## Анти-сбой: транскрипция на Windows

Нельзя останавливаться с сообщением `whisper не найден`, если команда `whisper` отсутствует в PATH. На этой машине локальный Whisper запускается не командой `whisper`, а по прямому пути:

```text
D:\soft\Whisper\whisper-cli.exe
```

В Git Bash/MSYS используй POSIX-путь:

```bash
/d/soft/Whisper/whisper-cli.exe
```

Модели лежат здесь:

```text
D:\soft\Whisper\models\ggml-base.bin
D:\soft\Whisper\models\ggml-small.bin
D:\soft\Whisper\models\ggml-large-v3.bin
```

Проверять нужно именно существование файла `D:\soft\Whisper\whisper-cli.exe`, а не наличие команды `whisper` в PATH.

Базовая команда:

```bash
/d/soft/Whisper/whisper-cli.exe \
  -m /d/soft/Whisper/models/ggml-base.bin \
  -t 4 -l ru -otxt -bs 1 -bo 1 -nf \
  -f /path/to/audio.wav
```

Если `base` даёт плохой русский текст, используй `small`, если модель есть:

```bash
/d/soft/Whisper/whisper-cli.exe \
  -m /d/soft/Whisper/models/ggml-small.bin \
  -t 4 -l ru -otxt -bs 1 -bo 1 -nf \
  -f /path/to/audio.wav
```

Если видео длинное, сначала нарежь аудио на сегменты и транскрибируй сегменты. Не пытайся бесконечно гонять один огромный WAV.

```bash
ffmpeg -y -i input.wav -f segment -segment_time 180 -c copy segments/part_%03d.wav
```

Если прямой локальный Whisper недоступен:

1. Проверь путь `D:\soft\Whisper\whisper-cli.exe` и модели.
2. Проверь альтернативный venv:

```text
D:\JetBrains\cl_maverick_kb\.venv_stt
```

3. Если `faster-whisper` доступен — используй его как fallback.
4. Если качественная транскрипция всё равно невозможна, не останавливай всю задачу: сделай минимум полезного результата по визуальным материалам, contact sheets, OCR/скриншотам и частичным аудио. Спорные бизнес-правила вынеси в `QUESTIONS.md`.

Запрещено завершать задачу формулировкой:

```text
Транскрипт не создан, потому что whisper CLI не найден, OPENAI_API_KEY не задан.
```

OpenAI API не является обязательным способом транскрипции для этого проекта. Его отсутствие не причина останавливать обработку.

Если агент запущен не на Windows-машине пользователя и физически не видит `D:\soft\Whisper` и исходное видео, нужно прямо сообщить: `задача запущена не в той среде`. Но если исходное видео и `D:\soft\Whisper` доступны — агент обязан продолжать автономно.

### 5. Выделить полезные факты

Из материала извлекай только то, что помогает выполнить задачу:

- точные пути в интерфейсе;
- названия кнопок, вкладок, отчётов, колонок;
- порядок действий;
- условия применения;
- ограничения;
- ошибки и диагностику;
- бизнес-смысл полей;
- риски и контрольные точки;
- поисковые слова пользователя;
- скриншоты, которые реально помогают инструкции.

Не переносить:

- разговорные повторы;
- внутренние обсуждения;
- неуверенные формулировки;
- персональные обращения;
- сырой текст транскрипта;
- фразы `по видео`, `в источнике`, `как сказала Настя`;
- неподтверждённые правила.

### 6. Проверить существующие страницы

Перед созданием новой страницы ищи существующие:

- по названию темы;
- по продукту;
- по пользовательской задаче;
- по ключевым словам из источника;
- по смежным страницам.

Если страница уже есть:

1. Прочитай её целиком.
2. Сравни новый материал со старой страницей.
3. Определи:
   - что добавить;
   - что уточнить;
   - что заменить;
   - что противоречит старому тексту;
   - какие скриншоты улучшат страницу.
4. Обнови каноническую страницу.

Новую страницу создавай только если это новая самостоятельная задача.

### 7. Написать инструкцию в vault

Сначала обновляется Obsidian vault:

```text
D:\JetBrains\cl_maverick_kb\Maverick_KB\wiki
```

Скриншоты для vault:

```text
D:\JetBrains\cl_maverick_kb\Maverick_KB\media
```

Публичная страница должна быть чистой инструкцией:

- краткое описание;
- для кого;
- когда применяется;
- что получится;
- перед началом;
- порядок действий;
- проверка результата;
- важные риски;
- частые ошибки;
- связанные страницы.

Пустые разделы удалять.

### 8. Вынести неизвестные факты в QUESTIONS.md

Если факт нельзя подтвердить, не пиши его как истину.

Вопросы записываются сюда:

```text
D:\JetBrains\cl_maverick_kb\Maverick_KB\QUESTIONS.md
```

Формат:

```md
## <Тема>

- [[Путь/к/странице]]: какой точный вопрос нужно уточнить?
```

В публичной статье не должно быть фраз:

- `возможно`;
- `скорее всего`;
- `нужно уточнить`;
- `по видео видно`;
- `из источника следует`;
- `QUESTIONS`;
- `TODO`.

Если нужно временно дать рабочую формулировку, она должна быть честной и полезной, без ложной точности. Например: `Используй отчёт для проверки ...`, а детали расчёта вынеси в вопросы.

### 9. Обновить служебные файлы vault

После изменения wiki:

- обнови `Maverick_KB/INDEX.md`, если добавлена новая страница;
- обнови `Maverick_KB/CHANGELOG.md` с датой и кратким описанием;
- обнови `Maverick_KB/QUESTIONS.md`, если остались пробелы.

### 10. Синхронизировать Pages

Публичная копия лежит здесь:

```text
D:\JetBrains\cl_maverick_kb_pages\docs
```

Нужно:

1. Скопировать/синхронизировать изменённые страницы из vault в `docs/`.
2. Конвертировать Obsidian-вставки картинок:

```md
![[media/manager/file.jpg]]
```

в MkDocs Markdown:

```md
![описание](../assets/media/manager/file.jpg)
```

3. Скопировать картинки в:

```text
D:\JetBrains\cl_maverick_kb_pages\docs\assets\media
```

4. Конвертировать wikilinks `[[...]]` в обычные Markdown-ссылки.
5. Обновить `mkdocs.yml`, если добавлена новая страница в навигацию.

### 11. Проверить локальную сборку

Из папки Pages:

```bash
cd /d/JetBrains/cl_maverick_kb_pages
uvx --with mkdocs-material mkdocs build --strict
```

Ошибки сборки, битые ссылки и отсутствующие картинки недопустимы.

Дополнительно проверить:

- нет ли `[[...]]` в публичных `.md`;
- нет ли `TODO`, `QUESTIONS`, `high-risk`, `verified`, `по видео`, `в источнике`;
- все локальные Markdown-ссылки существуют;
- все картинки существуют;
- поиск находит страницу по реальным пользовательским словам.

### 12. Визуально проверить страницу

Открой локальную собранную страницу или live-страницу и проверь:

- страница читается как инструкция;
- скриншоты видны и поддерживают шаги;
- нет панели встречи, таскбара, лишних окон;
- таблицы и UI-элементы читаемы;
- нет сырого конспекта;
- нет внутренних заметок.

### 13. Commit, push, deploy

В Pages-репозитории:

```bash
cd /d/JetBrains/cl_maverick_kb_pages
git status --short
git add docs mkdocs.yml
git commit -m "docs: update Maverick KB content"
git push origin main
```

Потом проверить GitHub Actions:

```bash
gh run list -R ogelslamovuk/cl_maverick_kb --limit 3
```

Нужный deploy должен быть:

```text
completed success
```

### 14. Live-проверка

Проверь опубликованные URL:

- изменённые страницы возвращают HTTP 200;
- ключевые изображения возвращают HTTP 200;
- search index обновился;
- поисковые фразы находят страницу;
- публичный текст не содержит внутренних маркеров.

## Особые правила для скриншотов

Если материал учебный/STAGE и пользователь не сказал обратное, данные считаются фейковыми. Не редактируй скриншоты агрессивно.

Если редактирование всё же нужно:

- скрывай только конкретные чувствительные значения;
- оставляй заголовки, колонки, табы, поля и структуру формы читаемыми;
- не превращай скриншот в бесполезное пятно.

## Особые правила для финансовых тем

Темы денег, оплат, возвратов, НДС, сертификатов и бухгалтерии — повышенный риск.

Нельзя:

- выдумывать расчёты;
- утверждать налоговые правила без подтверждения;
- писать точные бухгалтерские трактовки, если источник их не подтверждает;
- менять смысл отчётов на основании догадки.

Можно:

- описать, где находится отчёт;
- какие колонки видны;
- для какой проверки отчёт используется;
- какие вопросы надо уточнить у владельца процесса.

## Что писать в финальном ответе пользователю

Финальный ответ должен быть коротким и проверяемым:

```text
Готово.

Создано/обновлено:
- ...

Где лежит:
- ...

Проверка:
- build: OK
- links/assets: OK
- deploy: success
- live URL: ...

Осталось уточнить:
- ...
```

Не писать обещания вместо результата.
