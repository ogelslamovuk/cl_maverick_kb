## Главная цель

На выходе должны быть рабочие инструкции, которые помогают сотруднику выполнить задачу:

- что открыть;
- что проверить;
- что нажать;
- что означает поле/отчёт/статус;
- как понять, что всё сделано правильно;
- какие ошибки бывают;
- что передавать дальше.

Не нужен пересказ видео, лекция, учебник или внутренний конспект.

## Перед началом любой большой обработки

Сначала сформулируй пользователю:

1. ожидаемый результат;
2. какие материалы будешь обрабатывать;
3. какие страницы/разделы, вероятно, будут затронуты;
4. какие проверки сделаешь.

После этого остановись и жди явного разрешения: `approve`, `давай`, `ок`.

До разрешения не делай транскрипцию, извлечение кадров, редактирование файлов, build, commit, push.


## Автономный режим без повторного approve

Если пользователь дал initial message из `docs/MAVERICK_KB_INITIAL_MESSAGE.md`, это уже считается явным разрешением на полный цикл обработки. В этом режиме агент не должен останавливаться после плана и ждать дополнительного `approve`.

В автономном режиме агент сам выполняет весь цикл:

1. проверяет исходную папку;
2. инвентаризирует материалы;
3. извлекает аудио/кадры;
4. делает транскрипцию через локальный Windows Whisper по прямому пути;
5. при сбоях применяет fallback-и;
6. пишет/обновляет wiki;
7. обновляет `QUESTIONS.md`, `INDEX.md`, `CHANGELOG.md`;
8. синхронизирует Pages;
9. запускает build;
10. проверяет ссылки, картинки, поиск;
11. делает commit/push;
12. ждёт GitHub Pages deploy;
13. проверяет live URL;
14. только потом отвечает пользователю.

Остановка допустима только если:

- исходная папка физически недоступна;
- агент запущен не на той машине и не видит `D:\JetBrains` / `D:\soft`;
- нет прав на запись в проект;
- команда может уничтожить данные или требует явного выбора пользователя.

Отсутствие команды `whisper` в PATH, отсутствие `OPENAI_API_KEY`, отсутствие Python-модуля `openai` или отсутствие `faster_whisper` не являются причинами остановки.

## Как обрабатывать источник

### Видео / screencast

Для видео обязательно учитывать аудио и экран вместе:

- экран показывает интерфейс;
- голос объясняет смысл, бизнес-правила, исключения, риски;
- отдельный кадр без аудио может быть неверно понят;
- отдельный транскрипт без экрана может потерять UI-контекст.

Нужно получить:

- транскрипт;
- ключевые кадры/contact sheet;
- скриншоты для публикации;
- список подтверждённых фактов;
- список вопросов.


## Анти-сбой: транскрипция на Windows

Нельзя останавливаться с сообщением `whisper не найден`, если команда `whisper` отсутствует в PATH. На этой машине локальный Whisper запускается не командой `whisper`, а по прямому пути:

```text
D:\soft\Whisper\whisper-cli.exe
```

В Git Bash/MSYS используй POSIX-путь:

```bash
/d/soft/Whisper/whisper-cli.exe
```

Модели лежат здесь:

```text
D:\soft\Whisper\models\ggml-base.bin
D:\soft\Whisper\models\ggml-small.bin
D:\soft\Whisper\models\ggml-large-v3.bin
```

Проверять нужно именно существование файла `D:\soft\Whisper\whisper-cli.exe`, а не наличие команды `whisper` в PATH.

Базовая команда:

```bash
/d/soft/Whisper/whisper-cli.exe \
  -m /d/soft/Whisper/models/ggml-base.bin \
  -t 4 -l ru -otxt -bs 1 -bo 1 -nf \
  -f /path/to/audio.wav
```

Если `base` даёт плохой русский текст, используй `small`, если модель есть:

```bash
/d/soft/Whisper/whisper-cli.exe \
  -m /d/soft/Whisper/models/ggml-small.bin \
  -t 4 -l ru -otxt -bs 1 -bo 1 -nf \
  -f /path/to/audio.wav
```

Если видео длинное, сначала нарежь аудио на сегменты и транскрибируй сегменты. Не пытайся бесконечно гонять один огромный WAV.

```bash
ffmpeg -y -i input.wav -f segment -segment_time 180 -c copy segments/part_%03d.wav
```

Если прямой локальный Whisper недоступен:

1. Проверь путь `D:\soft\Whisper\whisper-cli.exe` и модели.
2. Проверь альтернативный venv:

```text
D:\JetBrains\cl_maverick_kb\.venv_stt
```

3. Если `faster-whisper` доступен — используй его как fallback.
4. Если качественная транскрипция всё равно невозможна, не останавливай всю задачу: сделай минимум полезного результата по визуальным материалам, contact sheets, OCR/скриншотам и частичным аудио. Спорные бизнес-правила вынеси в `QUESTIONS.md`.

Запрещено завершать задачу формулировкой:

```text
Транскрипт не создан, потому что whisper CLI не найден, OPENAI_API_KEY не задан.
```

OpenAI API не является обязательным способом транскрипции для этого проекта. Его отсутствие не причина останавливать обработку.

Если агент запущен не на Windows-машине пользователя и физически не видит `D:\soft\Whisper` и исходное видео, нужно прямо сообщить: `задача запущена не в той среде`. Но если исходное видео и `D:\soft\Whisper` доступны — агент обязан продолжать автономно.

## Как выбирать действие со страницей

Перед созданием новой страницы всегда ищи существующую.

Варианты:

1. **Обновить существующую страницу** — если задача уже есть.
2. **Слить дубли** — если несколько страниц описывают одно и то же.
3. **Создать новую страницу** — только если это новая самостоятельная задача.
4. **Добавить FAQ/заметку** — если материал маленький.
5. **Вынести в QUESTIONS** — если фактов недостаточно.

Новая запись источника не означает, что нужна новая страница.

## Стиль публичной инструкции

Пиши на русском, коротко, по делу.

Хорошо:

```text
Открой Manager → Отчёты → Предоплата (билеты).
Выбери дату, на которую нужно проверить непогашенные билеты.
```

Плохо:

```text
В видео Настя показывает, что можно попробовать посмотреть этот отчёт, возможно он подходит...
```

Запрещено в публичных страницах:

- `по видео видно`;
- `в источнике сказано`;
- `как сказала Настя`;
- `возможно` как замена факта;
- `нужно уточнить` внутри статьи;
- `TODO`;
- `QUESTIONS`;
- `high-risk`;
- `verified` как внутренний статус;
- сырые фрагменты транскрипта;
- внутренние комментарии участников.

## Как работать с неизвестными фактами

Если факт не подтверждён:

1. Не публикуй его как истину.
2. Не заполняй пустоту общими словами.
3. Запиши вопрос в:

```text
D:\JetBrains\cl_maverick_kb\Maverick_KB\QUESTIONS.md
```

Формат:

```md
## <Тема>

- [[Раздел/Страница]]: какой точный вопрос нужно уточнить?
```

Публичная страница должна быть уверенной там, где факты подтверждены, и молчать там, где фактов нет.

## Информационная архитектура

Основные разделы:

- `Билеты и касса`;
- `Manager / back-office`;
- `Зал и ресторан`;
- `Киоск`;
- `Афиша и расписание`;
- `Настройки и налоги`;
- `Поддержка`.

Размещай страницу там, где пользователь будет её искать, а не там, где внутренне расположен модуль продукта.

Если тема пересекает несколько разделов, веди одну каноническую страницу и добавляй ссылки из других разделов.

## Скриншоты

Скриншоты должны помогать шагу.

Для STAGE/training материалов данные считаются фейковыми, если пользователь не сказал обратное. Не редактируй скрины агрессивно. Сохраняй структуру таблиц, форм, вкладок и колонок.

Если редактирование нужно:

- скрывай только конкретные чувствительные значения;
- не закрывай заголовки и структуру;
- не оставляй панели встречи, таскбар и лишние окна.

## Проверка перед финалом

Из Pages repo:

```bash
cd /d/JetBrains/cl_maverick_kb_pages
uvx --with mkdocs-material mkdocs build --strict
```

Проверь:

- build прошёл;
- нет битых ссылок;
- нет отсутствующих картинок;
- нет `[[...]]` в публичных `.md`;
- нет `TODO`, `QUESTIONS`, `high-risk`, `verified`, `по видео`, `в источнике`;
- поиск находит новые страницы по пользовательским словам;
- локальная/публичная страница визуально читаема.

Потом:

```bash
git status --short
git add docs mkdocs.yml
git commit -m "docs: update Maverick KB content"
git push origin main
gh run list -R ogelslamovuk/cl_maverick_kb --limit 3
```

Дождись `completed success` и проверь live URL.

## Что считать готовой работой

Работа готова только если:

- инструкция написана/обновлена;
- вопросы вынесены в `QUESTIONS.md`;
- vault обновлён;
- Pages обновлены;
- build успешен;
- ссылки и картинки проверены;
- commit/push выполнен;
- GitHub Pages deploy успешен;
- live-страницы открываются;
- финальный ответ содержит только подтверждённые факты.
